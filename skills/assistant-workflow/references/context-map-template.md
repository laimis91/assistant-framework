# Context Map Template

Generated by Code Mapper during Discover phase. Stored at `.claude/context-map.md`.
Other agents (Code Writer, Architect) read this instead of re-exploring the codebase.

Think of this as the "smart data structure" — if the map is right, the implementation becomes self-evident.

## When to generate
- Medium+ tasks during Discover phase
- Code Mapper produces this as its primary output
- Regenerated per-session (not persisted across sessions — codebase may have changed)

## Template

```markdown
# Context Map
Generated: [date] | Task: [1-sentence description]

## Entry Points
- [file:line] — [what it does, how it's invoked]
- [file:line] — [what it does, how it's invoked]

## Key Interfaces & Implementations
| Interface | Implementor(s) | Location |
|-----------|----------------|----------|
| [name] | [concrete class(es)] | [file:line] |

## Data Flow
[source] → [transform/service] → [destination]
- [describe the primary data path relevant to this task]

## Dependencies (relevant to task)
- [module/package] — [what it provides, version if relevant]

## Config & Environment
- [file] — [what it configures]
- [env var / appsettings key] — [what it controls]

## Test Locations
| Area | Test File | Type |
|------|-----------|------|
| [module/feature] | [path] | Unit / Integration / E2E |

## Conventions Observed
- [naming pattern, e.g. "Services end in *Service, registered as scoped"]
- [architecture pattern, e.g. "CQRS with MediatR handlers"]
- [test pattern, e.g. "xUnit + Moq, Arrange-Act-Assert"]
```

## Rules
- Keep it compact — this feeds into other agents' context windows
- Only map what's relevant to the current task, not the entire codebase
- File paths must be relative to project root
- If the codebase is too large, focus on the modules the task touches
- Add `.claude/context-map.md` to `.gitignore` — this is ephemeral, per-session only
