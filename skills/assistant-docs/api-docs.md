# API Documentation Generator

## Protocol

### Step 1: Discover API Surface

Scan the codebase for:
- **ASP.NET Core**: `MapGet/MapPost/MapPut/MapDelete`, `[HttpGet]` attributes, controller actions
- **Minimal APIs**: `app.Map*` calls in Program.cs or endpoint files
- **gRPC**: `.proto` files
- **Other**: any public interface that represents an API boundary

For each endpoint/method, extract:
- Route/path
- HTTP method (if applicable)
- Parameters (path, query, body) with types
- Return type and status codes
- Authentication/authorization requirements
- Request/response DTOs

### Step 2: Analyze Existing Docs

Check for existing documentation:
- `docs/api/` or similar directory
- Swagger/OpenAPI specs
- XML doc comments
- README sections about API usage

If existing docs exist, **update** them rather than creating new files.

### Step 3: Generate

For each API surface, produce:

```markdown
## [Endpoint/Service Name]

[One-line description derived from method name and context]

### Request

[Method] [Route]

| Parameter | Location | Type | Required | Description |
|---|---|---|---|---|
| id | path | int | yes | [description] |

**Body:**
```json
{
  "field": "example value from actual DTO"
}
```

### Response

**200 OK**
```json
{
  "field": "example derived from return type"
}
```

**Error Responses**
| Status | When |
|---|---|
| 404 | [resource] not found |
| 400 | Validation failed |
```

### Step 4: Verify

- Every documented endpoint exists in code
- Every parameter type matches the actual type
- Every example value is realistic for the field type
- Authentication requirements match middleware/attributes

### Output Location

- If the project has a `docs/` directory, put API docs there
- If the project has Swagger, enhance the XML comments instead
- If no docs directory, create `docs/api.md`
- Mark generated sections: `<!-- generated by assistant-docs: api -->`
