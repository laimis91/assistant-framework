## Fully Implementable Improvements

These work identically on both Claude Code and Codex CLI. No platform-specific code needed.

### FI-1: Restructure SKILL.md with Progressive Disclosure

**Problem:** Our `assistant-workflow` SKILL.md is ~500+ lines (well over 5,000 words). Claude Code recommends 1,500-2,000 words for SKILL.md body, 5,000 words max. This eats significant context on every workflow invocation, competes with code/conversation for context space, and buries information deep where it may never be reached.

**Solution:** Restructure skills using the three-tier progressive disclosure model:
```
SKILL.md          (~1.5-2k words — core workflow, phase overview, routing)
references/       (detailed templates, prompts, checklists — loaded on demand)
examples/         (working examples — loaded when agent needs them)
scripts/          (automation scripts — executed, not loaded into context)
```

**Why it works on both platforms:** Both Claude Code and Codex CLI use identical SKILL.md format with `references/`, `examples/`, `scripts/` subdirectories. Progressive disclosure is built into both skill systems — metadata always loaded, body on trigger, resources on demand.

**Current state:** We already have `references/` in workflow but the SKILL.md itself is monolithic. The core SKILL.md should become a concise router pointing to references, not containing everything inline.

**Impact:** Saves ~3,000 tokens per workflow invocation. Frees context for actual code.

**Effort:** Medium (restructure ~8 skill files, move content to references/).

**Applies to:** assistant-workflow (biggest), assistant-review, assistant-onboard, assistant-research, assistant-security.

### FI-2: Improve Trigger Descriptions

**Problem:** Our skills use `triggers:` with regex patterns and `priority:` numbers for deterministic routing via `skill-router.sh`. The `description:` field is used by both platforms to show skill metadata and for implicit skill matching. Our descriptions could be more specific about trigger conditions.

**Solution:** Rewrite skill `description:` fields following Claude Code's recommended pattern: third-person language with specific user phrases. Current example:
```yaml
description: "Memory management for cross-session learning. Captures user preferences..."
```
Better:
```yaml
description: "This skill manages cross-session memory in the knowledge graph. Use when the user says 'remember this', 'save insight', 'what do you know about me', 'forget', 'update memory'. Also activates when the user provides a correction or states a behavioral preference."
```

**Why it works on both platforms:** Both platforms inject description into system prompt for skill matching. More specific descriptions improve matching accuracy on both.

**Impact:** Better skill triggering accuracy, fewer false matches.

**Effort:** Small (rewrite ~20 description fields).

### FI-3: Use `additionalContext` in SessionStart Hook Output

**Problem:** Our `session-start.sh` outputs plain text that gets injected into context. Claude Code's protocol supports returning structured JSON with `additionalContext` field, which is specifically designated as "developer guidance" rather than conversational text.

**Solution:** Return JSON from SessionStart hooks:
```json
{
  "additionalContext": "ACTIVE TASK JOURNAL:\n...\n---\nMEMORY RULES:\n...\n---\nSESSION START PROTOCOL:\n..."
}
```

**Why it works on both platforms:** Both Claude Code and Codex CLI accept `additionalContext` from SessionStart hooks. Plain text also still works as fallback. Gemini uses `{"additionalContext": "..."}` natively (we already do this for Gemini).

**Current state:** We already return JSON for Gemini but plain text for Claude/Codex. Unifying to JSON with `additionalContext` standardizes the output.

**Impact:** Cleaner context injection, proper separation of developer guidance from conversational text.

**Effort:** Small (update session-start.sh output format).

### FI-4: Use `updatedInput` for Tool Enforcement

**Problem:** We have `workflow-guard.sh` on PreToolUse that warns when the orchestrator edits files directly. We could go further — modifying tool inputs to enforce conventions.

**Solution:** PreToolUse hooks can return `updatedInput` to modify tool parameters before execution:
- Auto-add `--tl:on -v:minimal` to `dotnet build` commands
- Auto-add `--tl:on` to `dotnet test` commands
- Enforce file encoding or line endings
- Redirect writes to safe locations when in dry-run mode

**Why it works on both platforms:** Both Claude Code and Codex CLI support `updatedInput` in PreToolUse hook output. The modified input replaces the original before tool execution.

**Impact:** Consistent build flags, convention enforcement without relying on agent memory.

**Effort:** Small (add input modification logic to workflow-guard.sh or new hook).

### FI-5: Split Framework into Plugins

**Problem:** V1 is a monolithic framework — one install installs everything. Users who only want workflow + review don't need unity-*, assistant-creative, etc. Every skill description eats context tokens even if unused.

**Solution:** Restructure into focused plugins:

| Plugin | Skills | Purpose |
|---|---|---|
| `assistant-core` | memory, telos, explain, clarify | Foundation: memory graph, identity, basics |
| `assistant-dev` | workflow, review, tdd, security, onboard, docs | Development workflow |
| `assistant-research` | research, investigate, verify, web, thinking | Information gathering |
| `assistant-creative` | compose, creative, ideate, summarize, extract, learn | Content and creativity |
| `assistant-unity` | All unity-* skills | Game development (optional) |

**Why it works on both platforms:** Both have plugin systems:
- Claude Code: `plugin.json` manifest + auto-discovery
- Codex CLI: `.codex-plugin/plugin.json` manifest + marketplace system
- Both scan `skills/*/SKILL.md` for skill discovery within plugins

**Impact:** Lighter context (only load what's needed), independent versioning, potential marketplace distribution.

**Effort:** Large (restructure directories, create manifests, update install.sh).

### FI-6: Proper Hook Output Protocol

**Problem:** Our hooks dump everything as plain text or use ad-hoc JSON. Claude Code's protocol has specific fields with semantic meaning that we could use properly.

**Solution:** Standardize hook output across all scripts:

| Field | Purpose | When to Use |
|---|---|---|
| `additionalContext` | Guidance for the agent (invisible to user) | SessionStart, PostToolUse |
| `systemMessage` | Warning shown to user | Any event |
| `decision: "block"` | Prevent action | Stop, PreToolUse |
| `suppressOutput` | Hide hook output from transcript | When injecting context silently |
| `permissionDecision` | Allow/deny tool use | PreToolUse |

**Why it works on both platforms:** The JSON output schema is shared. Both platforms parse the same fields.

**Impact:** Cleaner separation of agent guidance vs user-visible messages.

**Effort:** Small (update JSON output in ~7 hook scripts).

## Conditional Improvements

These leverage Claude Code features that Codex doesn't support. Implementation requires platform-specific settings files (which install.sh already generates separately).

### CI-1: Prompt Hooks for Enforcement

**Problem:** Our enforcement hooks (`workflow-enforcer.sh`, `workflow-guard.sh`, `learning-signals.sh`) use bash scripts with grep/regex to detect patterns. This is brittle — can't understand intent, misses paraphrasing, produces false positives.

**What Claude gets:** `type: "prompt"` hooks where a fast Claude model evaluates the situation using natural language:
```json
{
  "type": "prompt",
  "prompt": "The agent is about to edit a file. Check if there's an active task journal at .claude/task.md with status BUILDING. If so, is the main agent editing directly instead of delegating to a code-writer subagent? If yes, warn. Context: $TOOL_INPUT",
  "timeout": 15
}
```

**What Codex gets:** Same bash `type: "command"` scripts as today. Identical functionality, just less nuanced detection.

**How to implement:** `claude-settings.json` gets prompt hooks for enforcer/guard/signals. `codex-settings.json` keeps command hooks. install.sh already generates separate settings files per agent.

**Why Claude-only:** Codex CLI only supports `type: "command"` hooks. No prompt or agent types.

**Impact:** Higher accuracy enforcement with fewer false positives on Claude. Same baseline on Codex.

**Effort:** Medium (write prompt hook definitions, test alongside bash fallbacks).

### CI-2: `if` Conditions on Hooks

**Problem:** All our hooks fire on every matching event and self-filter internally (checking for task journals, statuses, etc.). This means spawning a bash process even when irrelevant.

**What Claude gets:** Conditional hook execution:
```json
{
  "matcher": "Edit|Write",
  "hooks": [{
    "type": "command",
    "command": "workflow-guard.sh",
    "if": "Bash(dotnet *)"
  }]
}
```

**What Codex gets:** Hooks fire always, self-filter via bash (current behavior). Matcher regex is supported but not `if` conditions.

**How to implement:** Add `"if"` to claude-settings.json hook entries. codex-settings.json omits the field.

**Why Claude-only:** Codex doesn't support `if` conditions. The matcher (regex on tool name) is supported on both.

**Impact:** Reduced hook overhead on Claude — skip spawning processes for irrelevant events.

**Effort:** Small (add `if` fields to claude-settings.json template in install.sh).

### CI-3: Agent Hooks for Review Enforcement

**Problem:** `stop-review.sh` uses bash `grep` to check for review strings in the task journal. It checks "does the string `### Spec Review` exist?" — not "was the review actually thorough?"

**What Claude gets:** `type: "agent"` hook — a multi-turn agentic hook with file access that reads the task journal, evaluates review completeness, checks that findings were addressed, and decides to block with specific instructions.

**What Codex gets:** Same bash string-matching as today via `stop-review.sh`.

**Why Claude-only:** Codex doesn't support agent-type hooks.

**Impact:** Qualitative review enforcement (was the review good?) vs quantitative (does the string exist?).

**Effort:** Medium (design agent hook prompt, test extensively to avoid false blocks).

### CI-4: `asyncRewake` for Background Validation

**Problem:** Some validation is slow (full build, security scan, complexity analysis). Currently these block the agent until complete.

**What Claude gets:** `asyncRewake: true` hooks run in the background and wake the agent if they find a problem (exit code 2):
```json
{
  "type": "command",
  "command": "background-build-check.sh",
  "asyncRewake": true
}
```

**What Codex gets:** Not available. Agent runs synchronously. Validation must be inline.

**Why Claude-only:** Codex doesn't support async hook execution modes.

**Impact:** Non-blocking validation on Claude — agent continues working while checks run in background.

**Effort:** Medium (design background validation scripts, handle wake-up messaging).

### CI-5: `PostToolUseFailure` Hook

**Problem:** When a tool fails (build error, test failure, permission denied), the agent sees the error but we have no hook-level reaction. We could inject targeted guidance on failure.

**What Claude gets:** `PostToolUseFailure` hook fires with error details:
```json
{
  "tool_name": "Bash",
  "error": "NETSDK1100: ...",
  "error_type": "execution_error"
}
```
Hook returns guidance: "This is a Windows targeting error. Try adding `/p:EnableWindowsTargeting=true`."

**What Codex gets:** Not available. Codex has PostToolUse but not the failure variant. Agent handles errors via its own reasoning.

**Why Claude-only:** Event doesn't exist in Codex.

**Impact:** Faster error recovery — inject domain-specific fix suggestions immediately on failure.

**Effort:** Small (write failure handler script, register for PostToolUseFailure).

### CI-6: `additionalContext` on PostToolUse

**Problem:** After tool execution, we don't inject any guidance. We could add context like "build succeeded — proceed to next step" or "tests passed — 3 new, 0 regression."

**What both get:** PostToolUse hooks with `additionalContext` in output. Both platforms support this.

**Why conditional:** Claude can use `if` conditions to only fire on specific commands (e.g., `dotnet test`). Codex fires on all PostToolUse events and must self-filter.

**Impact:** Contextual guidance after tool execution on both platforms, with smarter filtering on Claude.

**Effort:** Small (write PostToolUse handler, add filtering logic).

## Skipped Improvements

These were evaluated and deliberately excluded. Documented here for future reference.

### SK-1: HTTP Hooks for External Integration

**What:** `type: "http"` hooks POST to external URLs (Slack, ClickUp, dashboards).

**Why skip:** Codex doesn't support it. The same integration is better achieved via MCP servers, which both platforms support. For example, we already have ClickUp MCP. Adding HTTP hooks would create a Claude-only integration path when MCP is universally supported.

**Revisit when:** If a use case requires real-time webhook-style integration that MCP can't serve (unlikely).

### SK-2: `once` Flag for One-Shot Hooks

**What:** A hook with `"once": true` runs once then auto-removes.

**Why skip:** Codex doesn't support it. Trivially simulated in bash with a flag file (`touch /tmp/.hook-ran-once; [[ -f /tmp/.hook-ran-once ]] && exit 0`). Not worth a framework feature for a one-line workaround.

**Revisit when:** Never — bash workaround is sufficient.

### SK-3: `CLAUDE_ENV_FILE` for Persistent Environment Variables

**What:** Hooks write `export VAR=value` to `$CLAUDE_ENV_FILE`, available in subsequent Bash tool calls.

**Why skip:** Claude-only. Our hooks don't need persistent env vars — they read state from files (task.md, session.md, graph.jsonl). The use case (direnv-style env loading) doesn't apply to our framework.

**Revisit when:** If a hook needs to persist environment state across tool calls (e.g., setting PATH for a specific SDK).

### SK-4: `FileChanged` / `CwdChanged` Reactive Hooks

**What:** Events that fire when files change or the working directory changes.

**Why skip:** Only Claude supports these (Codex doesn't). The primary use case (direnv integration, auto-reload on config change) isn't relevant to our framework. We don't watch for file changes — our hooks are event-driven around the agent lifecycle, not the filesystem.

**Revisit when:** If we need file-watching for auto-test-run or auto-build triggers.

### SK-5: Replace skill-router.sh with Prompt Hook

**What:** Use an LLM-based prompt hook instead of bash regex routing for skill matching.

**Why skip:** Deterministic regex routing is actually *better* for skill matching. It's predictable, debuggable, fast, and produces consistent results. An LLM-based router would introduce non-determinism — the same input might route to different skills on different runs. Our current `skill-router.sh` with regex patterns and priority ordering is the correct approach.

**Revisit when:** Never — deterministic routing is a design principle, not a limitation.

### SK-6: Plugin Marketplace Distribution

**What:** Package the framework for public distribution via Codex marketplace or similar.

**Why skip:** Premature. Our framework is personal/small-team. Marketplace adds packaging overhead, version management, compatibility testing, and documentation burden without clear benefit. The current `install.sh` approach works well for personal use.

**Revisit when:** If the framework grows beyond personal use and other developers want to install it.

## Claude Code Architecture Insights

Key architectural patterns discovered that inform our improvements.

### Hook System (27 events, 4 types)

- **Events fire at precise lifecycle points.** 9 core events cover the full agent lifecycle. 18 additional events cover file changes, permissions, config, worktrees, and more.
- **Parallel execution within batches.** All matching hooks for an event run in parallel via `Promise.all()`. Each hook has individual timeout (default 60s).
- **Hook priority:** Settings hooks > Plugin hooks > Definition order.
- **Aggregation:** If ANY hook blocks → aggregated `preventContinuation: true`. Permission: `deny` > `ask` > `allow` > `passthrough`.

### Skill System (Progressive Disclosure)

- **Three-tier loading:** Metadata (always, ~100 words) → Body (on trigger, ~2k words ideal) → Resources (on demand, unlimited).
- **Description is the trigger.** No regex patterns — Claude matches user intent against description text.
- **Skill descriptions capped at 250 chars** in the `/skills` menu. Keep descriptions concise.
- **MCP tool descriptions capped at 2KB** per tool. Prevents context bloat from verbose tool schemas.

### Context Management

- **Compaction thrash detection:** After 3 consecutive compactions that refill to limit immediately, Claude Code stops with an error instead of burning API calls.
- **Read tool deduplication:** Re-reading unchanged files produces compact output.
- **Token estimation fixes:** Thinking blocks and tool_use blocks were over-counted, causing premature compaction (fixed in v2.1.85).
- **Image preservation during compaction** for prompt cache reuse (v2.1.84).

### Memory System (Auto-Memory)

- **Location:** `~/.claude/MEMORY.md` + `~/.claude/memories/*.md` (auto-saved after sessions).
- **Limits:** 25KB + 200 lines for memory index. Truncated beyond this.
- **Configurable:** `autoMemoryDirectory` setting for custom storage location.
- **Timestamps:** Memories include last-modified timestamps for freshness reasoning (v2.1.82).

Note: This is Claude Code's built-in auto-memory. Our framework's memory-graph MCP is a separate, richer system that coexists with it.

## Codex CLI Architecture Insights

Key architectural patterns that constrain our cross-platform compatibility.

### Hook System (5 events, 1 type)

- **Only 5 events:** SessionStart, PreToolUse, PostToolUse, UserPromptSubmit, Stop.
- **Only `command` type.** No prompt, agent, or HTTP hooks.
- **PreToolUse limited to Bash tool only** (currently).
- **No `if` conditions, `async`, `asyncRewake`, or `once` flags.**
- **Matcher is regex-based** (similar to Claude Code's matcher field).
- **Default timeout 600s** (10x Claude Code's 60s default).

### Skill System (Compatible)

- **Same SKILL.md format** with YAML frontmatter (name, description, version).
- **Same progressive disclosure** model with references/, examples/, scripts/, assets/.
- **Hierarchical scan:** `.agents/skills` in cwd → parent → repo root → user home → admin → system.
- **Activation modes:** Explicit (`/skills` or `$skill-name`) and implicit (description-based auto-matching).
- **Built-in skills:** `$skill-creator`, `$skill-installer`, `$plugin-creator`.

### Agent System

- **Built-in types:** default, worker, explorer.
- **Custom agents:** TOML files in `~/.codex/agents/` or `.codex/agents/`.
- **Feature flag:** `features.multi_agent = true` required.
- **Limits:** `max_threads: 6`, `max_depth: 1` (no recursive spawning), `job_max_runtime_seconds: 1800`.

### AGENTS.md (vs CLAUDE.md)

- **Discovery chain:** `AGENTS.override.md` (highest priority) → `AGENTS.md` → fallback filenames.
- **Scope chain:** Global (`~/.codex/`) → project root to cwd, walking each directory.
- **Size limit:** 32 KiB total (`project_doc_max_bytes`).
- **Injection:** Each file becomes a separate user-role message.

### Plugin System

- **Manifest:** `.codex-plugin/plugin.json` with name, version, description, skills, mcpServers, apps.
- **Marketplace:** JSON catalogs at official, repo, or personal levels. Installation policies per plugin.
- **Cache:** `~/.codex/plugins/cache/$MARKETPLACE/$PLUGIN/$VERSION/`.
- **Public publishing not yet available.**

## Implementation Strategy

### Phase 1: Quick Wins (Small effort, cross-platform safe)

1. **FI-2:** Improve trigger descriptions across all 20+ skills
2. **FI-3:** Use `additionalContext` in SessionStart hook output
3. **FI-6:** Standardize hook output protocol (JSON fields)
4. **FI-4:** Use `updatedInput` for dotnet build/test flag enforcement
5. **CI-2:** Add `if` conditions to Claude-only settings (no Codex change needed)
6. **Metrics enforcement:** Complete the pending stop-hook implementation

### Phase 2: Structural Improvements (Medium effort, high impact)

7. **FI-1:** Restructure SKILL.md files for progressive disclosure (start with workflow, review, onboard)
8. **CI-1:** Add prompt hooks for workflow-enforcer and workflow-guard (Claude-only enhancement)
9. **CI-5:** Add PostToolUseFailure hook for error guidance (Claude-only)
10. **CI-6:** Add PostToolUse context injection (cross-platform with conditional filtering)

### Phase 3: Architecture Evolution (Large effort, long-term)

11. **FI-5:** Split framework into plugins (core, dev, research, creative, unity)
12. **CI-3:** Agent hooks for qualitative review enforcement (Claude-only)
13. **CI-4:** asyncRewake for background validation (Claude-only)

### Guiding Principle

**Progressive enhancement.** Bash command hooks are the baseline (works everywhere). Claude-specific features layer on top via separate settings files. The install.sh architecture already handles this split — `claude-settings.json` and `codex-settings.json` are generated independently.

Think of it like web development: HTML (bash hooks) works in every browser. CSS (prompt hooks, `if` conditions) enhances the experience in modern browsers. JavaScript (agent hooks, asyncRewake) adds interactive features where supported.
